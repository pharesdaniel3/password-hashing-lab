1. Why Hash Passwords?

Storing plain text passwords is extremely dangerous. If a database is breached, attackers immediately get all user credentials.

Hashing converts a password into a fixed-length string using a one-way function. It cannot be reversed to get the original password.

2. Basic Hashing (Insecure - No Salt)

Try the same password multiple times. Notice how the hash is always identical.

⚠️ Vulnerability: Identical passwords produce identical hashes → Easy rainbow table attacks.

3. Rainbow Table Attack Simulation

Attackers use precomputed tables of common passwords and their hashes.

4. Secure Hashing with Salt

Each password gets a unique random salt. Hash the same password multiple times.

Key Point: Same password + different salt = completely different hash. This defeats precomputed attacks.

Key Takeaways

  • ✅ Never store passwords in plain text
  • ✅ Always use a unique random salt per user
  • ✅ Use slow, memory-hard algorithms (PBKDF2, bcrypt, Argon2id)
  • ✅ Salting makes rainbow tables useless
  • ✅ Best practice in 2026: Argon2id
